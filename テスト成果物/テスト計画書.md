# テスト計画書

## 1. テスト目的

本テストは、個人向けタスク管理 Web アプリ「タスクマネージャー」について、利用者がタスクの計画、実績記録、定常タスク管理、カレンダー確認、分析、設定を継続的に安全かつ正確に利用できることを確認するために実施する。

特に、予定時間と実績時間の比較、中断/再開セッションを含む実働時間計測、Supabase 認証とユーザー別データ分離、タスク・カテゴリ・定常タスクの CRUD、分析グラフの集計結果について、主要なユーザーフローと高リスク領域を中心に信頼性を確認する。

本計画は、公開リポジトリ上の README、画面一覧、機能一覧、実装コード、Supabase マイグレーションを根拠にした暫定テスト計画である。仕様が明文化されていない箇所は推測で確定せず、未決事項として管理する。

## 4. テストスコープ

### 4.1 対象範囲

- README、画面一覧、機能一覧に記載された主要機能 F001〜F505、共通機能 F601〜F702。
- 認証後の主要ユーザーフロー: サインアップ/ログイン、カテゴリ作成、タスク作成、実績記録、中断/再開、完了、分析確認、エクスポート。
- タスクデータの永続化、ユーザーごとのデータ分離、RLS による他ユーザーデータ不可視性。
- Chrome 最新版、Edge 最新版での基本動作。
- `npm run build`、`npm run typecheck`、`npm run lint` による静的品質確認。
- Supabase migrations を適用した DB スキーマの整合性確認。

### 4.2 対象外/制限付き範囲

- スマートフォン表示は README で未保証とされているため、正式な合否判定対象外とする。ただし、重大な画面崩れの探索的確認は実施候補とする。
- 本番運用監視、バックアップ/リストア、障害復旧、課金、監査ログは仕様未記載のため対象外または要確認。
- Supabase 自体の可用性、メール配送、外部サービス障害はアプリ単体テストの対象外。ただし、エラー時の表示・復帰性は確認対象とする。
- 通知機能の詳細なタイミング、許可拒否時の期待挙動は未確定のため、計画段階では要確認扱いとする。
- 既存自動テストが確認できないため、回帰テスト資産の実行は現時点では対象外。代替として新規テスト設計・実装対象を計画する。

## 9. 前提条件

- テストは公開リポジトリの `main` ブランチ相当を対象とする。
- Supabase URL と anon key は公開リポジトリに含めず、テスト環境で別途設定する。
- DB migrations は README 記載順に適用されていることを前提とする。
- 本計画書ではスマートフォン表示を正式な合否対象に含めない。
- 既存の自動テストは確認できていないため、新規テスト設計時に自動化対象を選定する。

## 10. 正式テスト資産/回帰資産

現時点で、リポジトリ上に既存の Unit test、Integration test、E2E test、CI 定義、ゴールデンデータ、受入テスト仕様は確認できていない。

そのため、本テストプロセスでは以下を計画する。

- 既存テスト資産の再確認。
- `npm run build`、`npm run typecheck`、`npm run lint` を最低限の静的回帰確認として扱う。
- 高リスク領域である認証/RLS、タスク CRUD、中断/再開、分析集計、定常タスクを新規自動テスト候補とする。
- 人間による確認が必要な通知許可、ブラウザ差異、見た目、エクスポートファイル確認を手動/探索的テスト候補とする。

## 11. 後続成果物の品質ゲート

- テスト分析では、少なくとも R001、R002、R003、R006、R007、R012、R013、R015 の高リスク領域について詳細な観点を作成する。
- テスト設計では、日時境界、状態遷移、DB/RLS、複数ユーザー、複数セッション、空データ、大量データ、削除時参照関係を明示的に扱う。
- テストケースでは、コードベース自動、E2E 自動、人間実行、質問待ちを分離する。
- テスト実装前に、テスト環境とテスト用 Supabase の利用可否を確認する。
- 仕様未確定の項目は質問待ちとして残し、推測で合否基準を固定しない。

| Q001 | テスト対象の本番 URL またはローカル実行 URL が未指定 | E2E 実行範囲が確定できない | テスト対象 URL を指定する |
| Q002 | テスト用 Supabase プロジェクト、テストユーザー、初期データ、リセット手順が未指定 | 再現性のある統合/E2E テストが困難 | テスト専用環境を用意する |
| Q003 | `.env.example` が存在しない | 環境構築で迷いやすい | `.env.example` またはセットアップ補足を追加する |
| Q004 | 通知機能の期待タイミング、許可拒否時の合否基準が未定義 | 通知テストの判定が曖昧 | 通知仕様を明文化する |
| Q005 | 分析集計の丸め、対象期間、未完了/中断タスクの扱いが未定義 | 分析結果の合否判定が曖昧 | 集計ルールを仕様化する |
| Q006 | 日時・タイムゾーン・日跨ぎタスクの扱いが未定義 | カレンダー/実績/分析でズレる可能性 | JST 前提などの基準を決める |
| Q007 | Edge での E2E 実行要否、スマートフォン確認範囲が未確定 | 互換性テスト範囲が曖昧 | 対象ブラウザ/端末範囲を明確化する |
| Q008 | 受入者、合格閾値、P2 欠陥の扱いが未指定 | 終了判断が曖昧 | 個人開発向けの暫定終了基準を決める |
| Q009 | 既存自動テスト/CI が存在しない | 回帰検出力が低い | 最小限の Unit/E2E/CI 方針を決める |
| Q010 | 大量データ時の性能目安が未定義 | 性能テストの合否が曖昧 | 目安件数と許容応答時間を決める |
