# テスト計画書

## 1. テスト目的

本テストは、個人向けタスク管理 Web アプリ「タスクマネージャー」について、利用者がタスクの計画、実績記録、定常タスク管理、カレンダー確認、分析、設定を継続的に安全かつ正確に利用できることを確認するために実施する。

特に、予定時間と実績時間の比較、中断/再開セッションを含む実働時間計測、Supabase 認証とユーザー別データ分離、タスク・カテゴリ・定常タスクの CRUD、分析グラフの集計結果について、主要なユーザーフローと高リスク領域を中心に信頼性を確認する。

本計画は、公開リポジトリ上の README、画面一覧、機能一覧、実装コード、Supabase マイグレーションを根拠にした暫定テスト計画である。仕様が明文化されていない箇所は推測で確定せず、未決事項として管理する。

### 3.1 分析集計仕様

- 集計単位は単独タスクと末端の子タスクとし、子タスクを持つ親タスクは除外する。`track_actual = false` はステータス件数・完了率に含め、実績時間・所要時間差異分析から除外する。
- 正式な予定工数は `quantity × time_per_unit` とする。0 または未入力は「予定工数未入力」として予定時間集計から除外し、予定開始〜終了の長さを代用しない。
- 実績工数は、セッション合計、`actual_time`、`actual_start`〜`actual_end` の順で採用する。予定工数は代用しない。未着手は 0 分、進行中は終了済みセッションと開いているセッションの現在時刻まで、中断中は終了済みセッションだけ、完了はセッション合計を優先する。実績情報がなければ「実績未入力」として実績集計から除外する。
- 件数・完了率は予定開始月に所属させる。予定工数は予定区間が対象期間と重なる比率、実績工数はセッションまたは実績区間が対象期間と重なる分だけを日別・月別へ計上する。
- 内部計算はミリ秒または小数分を保持し、タスク単位・日単位では丸めない。表示時だけ時間を小数第1位、割合を整数％へ四捨五入し、テスト期待値は丸め前の分で判定する。
- 所要時間差異は実績と予定の差が +1 分超なら超過、-1 分未満なら短縮、±1 分以内なら一致とする。超過・短縮要因は分類条件ではなく補足情報として扱う。

### 3.2 日時・タイムゾーン仕様

- 業務タイムゾーンは端末設定に関係なく `Asia/Tokyo`（JST、UTC+9）とし、DSTは対象外とする。DBにはUTCのISO日時を保存し、入力はJSTとして解釈、表示と業務判定はJSTで行う。
- 日単位は `[当日00:00 JST, 翌日00:00 JST)`、月単位は `[月初00:00 JST, 翌月月初00:00 JST)` の半開区間とする。開始時刻を含み終了時刻は含めない。
- 日跨ぎ・月跨ぎタスクは各日・各月との重複分だけ予定・実績へ計上する。終了が00:00の場合は終了日の占有時間としてカレンダー表示しない。完了率・件数はJSTの予定開始月へ所属させる。
- タスク一覧の日付フィルターはJST対象日と1分以上重なるタスクを対象とする。予定終了日フィルターは終了日時をJST表示した暦日で判定し、「今日」もJST日付を使用する。
- 定常タスクの曜日、期間開始日、期間終了日はJST暦日とし、期間の両端を含む。日跨ぎは「翌日終了」で明示し、未選択時は終了時刻が開始時刻以下なら入力エラーとする。
- 境界期待値として23:59/00:00/00:01、月末、年末年始、2月28日/29日、端末タイムゾーンJST/UTC/米国、終了00:00、日跨ぎセッション、月跨ぎ按分、定常タスク生成を固定する。

## 4. テストスコープ

### 4.1 対象範囲

- README、画面一覧、機能一覧に記載された主要機能 F001〜F505、共通機能 F601〜F702。
- 認証後の主要ユーザーフロー: サインアップ/ログイン、カテゴリ作成、タスク作成、実績記録、中断/再開、完了、分析確認、エクスポート。
- タスクデータの永続化、ユーザーごとのデータ分離、RLS による他ユーザーデータ不可視性。
- Chrome 最新版、Edge 最新版での基本動作。
- Galaxy S20 5G相当（Android Chrome最新版、CSS viewport 360×800、縦向き）での主要画面・主要操作。
- `npm run build`、`npm run typecheck`、`npm run lint` による静的品質確認。
- Supabase migrations を適用した DB スキーマの整合性確認。

### 4.2 対象外/制限付き範囲

- 横向き、折りたたみ端末、タブレット、および実機固有差は制限付き範囲とする。正式判定はGalaxy S20 5G相当の360×800エミュレーションで行い、主要なレスポンシブ境界は360pxと768px前後で確認する。
- 本番運用監視、バックアップ/リストア、障害復旧、課金、監査ログは仕様未記載のため対象外または要確認。
- Supabase 自体の可用性、メール配送、外部サービス障害はアプリ単体テストの対象外。ただし、エラー時の表示・復帰性は確認対象とする。
- ブラウザ通知はタブを開いている間のみを対象とし、Service Worker等によるバックグラウンド通知は対象外とする。
- 既存自動テストが確認できないため、回帰テスト資産の実行は現時点では対象外。代替として新規テスト設計・実装対象を計画する。

## 9. 前提条件

- テストは公開リポジトリの `main` ブランチ相当を対象とする。
- Supabase URL と anon key は公開リポジトリに含めず、テスト環境で別途設定する。
- DB migrations は README 記載順に適用されていることを前提とする。
- Galaxy S20 5G相当の360×800縦向きエミュレーションを正式な合否対象とする。
- 各テストは必要なユーザーとデータをシナリオ内で作成し、他のテストと識別できる一意な値を使用して終了時に削除する。
- 既存の自動テストは確認できていないため、全画面・全機能の主要経路を少なくとも1回通す範囲で新規自動テストを設計する。
