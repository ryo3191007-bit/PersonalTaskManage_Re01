## 2. アプリ概要

PersonalTaskManage_Re01 は、業務の予定・実績を管理する個人向けタスク管理Webアプリです。

主な機能:

- メール/パスワード認証
- タスク登録・編集・削除
- ステータス変更
- 中断/再開を含む作業セッション記録
- カレンダー表示
- 定常タスク生成
- カテゴリ管理
- 分析画面での予定/実績/差異集計
- ブラウザ通知
- アカウント情報更新
- タスク/カテゴリ一括削除

## 7. 最近決定・実装された重要仕様

### 7.1 通知仕様

- 開始通知: 予定開始時刻に、未着手タスクのみ通知。
- 終了通知: 予定終了時刻に、未完了タスクのみ通知。
- 完了済み・削除済みタスクは通知しない。
- 予定時刻変更時は古い通知を取消して再予約。
- 同じ通知を重複表示しない。
- アプリ起動時点ですでに過ぎた通知は表示しない。
- 予定開始・終了が未入力なら通知は作成しない。
- 通知本文にはタスク名と予定時刻を含める。
- 終了通知本文:
  - 未着手: `終了予定時刻です。まだ開始されていません`
  - 進行中: `終了予定時刻になりました`
  - 中断中: `終了予定時刻です。現在中断中です`

制約:

- 通知は `setTimeout` ベースのため、ページを閉じると通知されない。

### 7.2 分析集計仕様

実績時間:

- 予定時間を実績へ代用しない。
- 未着手: 実績0分。
- 進行中: 終了済みセッション + 現在進行中セッションを現在時刻まで。
- 中断中: 終了済みセッションのみ。
- 完了: セッション合計を優先。
- セッションなし: `actual_time`、次に `actual_start` ～ `actual_end`。
- 実績情報なし: 実績集計から除外。

予定時間:

- 正式な予定工数は `quantity × time_per_unit`。
- 予定開始/終了日時は、日別・月別按分に使う。
- 予定工数が0または未入力なら予定工数未入力扱い。
- 予定日時の長さを予定工数の代用品にしない。

集計対象:

- 子タスクを持つ親タスクは除外。
- 単独タスクと末端の子タスクを集計。
- `track_actual=false` は実績時間・所要時間差異分析から除外。
- ステータス件数・完了率には `track_actual=false` も含める。

丸め:

- 内部計算はミリ秒または小数分のまま。
- 表示時だけ時間を小数第1位へ四捨五入。
- 割合は表示時だけ整数％へ四捨五入。
- テスト期待値は丸め前の分で判定。

所要時間差異:

- `実績 > 予定 + 1分`: 超過
- `実績 < 予定 - 1分`: 短縮
- `±1分以内`: 一致

### 7.3 日時・タイムゾーン仕様

- 業務上の基準タイムゾーンは常に `Asia/Tokyo`。
- 端末のタイムゾーン設定に関係なくJSTで入力・表示・集計。
- DB保存はUTC ISO日時。
- 入力値はJSTとして解釈してUTCへ変換。
- 表示時は明示的に `Asia/Tokyo` を指定。
- 日単位は `[当日00:00 JST, 翌日00:00 JST)`。
- 月単位は `[月初00:00 JST, 翌月月初00:00 JST)`。
- 開始時刻は含み、終了時刻は含めない。
- 日跨ぎ・月跨ぎは重複分だけ按分。
- 終了が00:00の場合、その終了日の占有時間として表示しない。
- 定常タスクの曜日・期間開始・期間終了はJST基準。
- 定常タスクの期間開始日と終了日は両方含む。
- 日を跨ぐ定常タスクは「翌日終了」を明示する。

### 7.6 中断/再開セッションとカレンダー表示

2026-06-27時点のPR #9で以下を修正済みです。

- タスク編集画面の中断/再開入力は、入力欄の `blur` ではDB保存しない。
- 既存タスク編集のsubmit時に、フォーム上の中断/再開 entries から期待される `task_sessions` を組み立て、既存セッションの更新・新規作成・不要セッション削除を一括同期する。
- 中断ステータス登録時の `actual_time` は現在時刻ではなく、入力された実績帯に基づいて算出する。
- 中断中タスクの `suspended_at` は最後の中断時刻を保存する。
- 完了ステータスの新規登録では、実績開始/終了を手動編集していない限り、予定開始/終了を実績開始/終了へ同期する。
- `TaskContext.updateSession` は `session_end: null` を受け取れる。
- カレンダー日表示では、ゼロ長セッションまたは `session_start >= session_end` のセグメントを描画しない。
- カレンダー日表示の予定差分枠は、同一タスクにつき1回だけ描画する。
- カレンダー日表示のレーン計算は、タスク全体のspanではなく実作業セグメント単位に寄せている。中断コネクターは直前の実作業セグメントの列に合わせる。

### 9.1 開発サーバー起動

このCodex環境では通常のPowerShell PATHに `node` / `npm` がない場合があります。その場合は、CodexバンドルのNode.jsでViteを直接起動してください。

確認済みの起動コマンド:

```powershell
$env:CI = "true"
$wd = "C:\Users\ryo31\OneDrive\ドキュメント\PersonalTaskManage_テスト管理\PersonalTaskManage_Re01"
$node = "C:\Users\ryo31\.cache\codex-runtimes\codex-primary-runtime\dependencies\node\bin\node.exe"
Start-Process -FilePath $node `
  -ArgumentList @("node_modules\vite\bin\vite.js", "--host", "127.0.0.1", "--port", "5173", "--clearScreen", "false") `
  -WorkingDirectory $wd `
  -WindowStyle Hidden
```

起動確認:

```powershell
Invoke-WebRequest -Uri "http://127.0.0.1:5173/" -UseBasicParsing -TimeoutSec 5
```

`HTTP 200` が返れば、in-app browser の `http://127.0.0.1:5173/` を再読み込みしてください。Vite 8 は `CI` が文字列 `"true"` でない場合、非対話プロセスの stdin 終了を検知してすぐ終了することがあります。

### 9.2 OneDrive配下の `node_modules`

過去にOneDriveの削除確認ポップアップ後、Tailwindの `preflight.css` が見つからないViteエラーが発生しました。

症状例:

```text
[plugin:vite:css] [postcss] ENOENT: no such file or directory
... tailwindcss/lib/css/preflight.css
```

対応:

```powershell
Remove-Item -Recurse -Force node_modules
Remove-Item -Force package-lock.json
npm install
npm run dev
```

ただし、削除系コマンドはユーザー確認の上で実行してください。

### 9.3 `.env`

`.env` はGit管理対象外です。次スレッドで起動確認する場合、`.env` の存在と以下を確認してください。

```text
VITE_SUPABASE_URL
VITE_SUPABASE_ANON_KEY
```

キーの実値はチャットやPR本文へ書かないでください。

### 9.4 Supabase Free/Pro

SupabaseのLeaked Password ProtectionはPro以上が必要です。Freeプランでは警告が残る可能性があります。現時点では公開自体のブロッカーではなく、解消するならPro化または別途パスワードポリシー/注意喚起で補完します。

### 9.5 Bolt

修正作業はCodex中心で進めています。Boltは主に公開作業で使っていました。Bolt側DBをSupabaseへClaim済みの経緯があります。DB接続先やAPIキーが変わった場合は `.env` とSupabaseプロジェクトの対応を確認してください。

### 9.6 Codex / Agent運用方針

以後の作業では、Codexは司令塔・管理者としてふるまいます。作業単位ごとに目的、入力成果物、生成物、検証結果、残課題を管理し、必要に応じてローカルQwen、CodexバンドルPython/Node、検索、検証スクリプト、ブラウザ、GitHub連携などの各種ツールを使います。

トークン節約が必要な場合は、設計表や成果物の全量を会話へ貼らず、ローカルファイルをソースとして機械抽出・検証し、会話には集計結果と判断だけを返します。生成物はファイルへ保存し、レビューや整合性確認も可能な限りスクリプトで再計算します。
